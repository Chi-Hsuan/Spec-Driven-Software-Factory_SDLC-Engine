# Extract Requirement — Thinking Prompt

<!-- 只負責思考，不決定輸出格式。輸出格式屬於 Template；本檔案只能被 skill/extract-requirement 讀取。 -->

## Thinking Strategy

1. 先問「使用者想完成什麼任務」，而不是「系統要做什麼功能」——從使用者目標反推功能點，避免功能堆砌。
2. 對每個功能點，追問「這是必要的，還是使用者順口提到的？」，區分核心需求與附帶想法。
3. 對每個功能點，追問「怎麼知道它做完了？」，若答不出來，標記為待釐清的 Gap，不直接假設驗收標準。
4. 檢查是否有隱含的非功能性期待（例如「要快」「要安全」），即使使用者沒有明講。
5. 遇到矛盾陳述時，兩者都記錄下來並列為待澄清，不自動選一個當作正確答案。

---

## Reasoning Rules

- **可獨立驗證原則**：若無法想像一個具體情境來驗證某個功能點，代表描述還不夠具體，應標記為 Gap，而非勉強寫成一條需求。
- **功能 vs 非功能判準**：描述「使用者能做的事」歸類為 Functional；描述「系統的品質特性」歸類為 Non-Functional。
- **缺口優先原則**：同一個概念出現超過一種互相衝突的說法時，視為缺口，不自行選擇其中一種。
- **禁止腦補**：任何輸入原文沒有明確提到的細節，不得自行補上，必須列為 Gap。

---

## Quality Checklist

- [ ] 每個功能點都能回答「這個功能點的使用者是誰？」
- [ ] 若完全不認識這個產品，這段描述夠不夠讓人理解要做什麼？
- [ ] 有沒有哪裡是自己腦補出來的細節，而不是輸入原文明確提到的？
- [ ] 是否誤把使用者隨口提到的想法當成正式需求，沒有加以區分？
- [ ] 是否用技術術語（例如「做 CRUD」）取代了使用者實際使用的語言？

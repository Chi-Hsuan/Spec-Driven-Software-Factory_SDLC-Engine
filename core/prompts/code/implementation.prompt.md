# Implementation — Thinking Prompt

## Thinking Strategy

1. 先讀 task 的 `module_ref`，回頭定位到規格中對應的區塊，確認這個 task 真正要做的事。
2. 逐項核對規格描述的行為（輸入、輸出、狀態、錯誤情境），在動手寫程式前先在腦中模擬一次流程。
3. 遇到規格沒有明講的細節，先問「這是可以從既有規格合理推斷的，還是真的沒有定義」，屬於後者則停下回報。
4. 完成後，反過來用規格逐條檢查程式碼行為，而不是只憑印象覺得「應該對了」。

---

## Reasoning Rules

- **規格是唯一依據**：任何實作決策都要能指向規格中的具體描述，不可依賴「業界慣例」或「我覺得應該這樣」。
- **不擴大範圍**：只實作 task 要求的部分，不順便實作規格中尚未被排進 task 的其他功能。
- **樣式/資料一律引用**：Frontend 樣式引用 design-tokens.json 的 token path；Backend 資料結構引用 schema.sql 的欄位定義，皆不寫死數值。
- **遇到矛盾就停**：規格與 task 描述矛盾時停止並回報，不自行選擇一方為準。

---

## Quality Checklist

- [ ] 每個實作決策是否都能指回規格中的具體描述？
- [ ] 是否實作了規格未要求、task 也未涵蓋的額外行為？
- [ ] Frontend 是否有任何寫死的樣式數值？
- [ ] 是否有規格未定義、但已被靜默假設的邊界情況？

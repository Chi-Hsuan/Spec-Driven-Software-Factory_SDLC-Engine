# Refactor — Thinking Prompt

<!-- 只負責思考，不決定輸出格式。只能被 skill/refactor-from-spec 讀取。 -->

## Thinking Strategy

1. 先比對新舊規格版本，具體列出「哪裡不一樣了」，而不是憑印象猜測變了什麼。
2. 針對每個差異點，問「這會影響程式碼的哪個部分」，圈出實際需要調整的範圍。
3. 若涉及資料庫變更，先確認 migration-plan.md 是否已有對應的 migration_id 與 rollback 方式，沒有的話停下回報。
4. 調整完後，重新檢查是否有「順手改掉」但其實跟本次規格變更無關的程式碼。

---

## Reasoning Rules

- **差異驅動**：只處理規格差異涉及的範圍，不藉機做無關的重構或風格調整。
- **資料變更走正式流程**：資料庫層級的變更必須透過 migration，不可直接修改既有資料結構。
- **保留歷史脈絡**：implementation-notes.md 需記錄本次變更對應的規格版本，方便之後追溯。

---

## Quality Checklist

- [ ] 是否清楚列出了新舊規格的具體差異？
- [ ] 修改範圍是否僅限於差異涉及的部分？
- [ ] 資料庫變更是否透過正式 migration 流程執行？
- [ ] 是否混入了與本次規格變更無關的程式碼調整？
